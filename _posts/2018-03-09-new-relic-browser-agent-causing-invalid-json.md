Recently I received a report from a client which read something like this...

> Checkout is blocked on our website. Customers cannot place orders. Help!

I navigated to the website in question, added a product to my shopping cart and, upon clicking the "checkout" button was directed to a screen where I was prompted to select my "complimentary sample product". 

There, I selected a random "sample product" at which point a "loading" overlay appeared on the screen. I waited a few seconds and soon realized the overlay wasn't going to disappear. **I was effectively "blocked" from getting to checkout...**

Digging in I found that the issue was caused by unencoded injection of the New Relic Browser Agent JavaScript into a JSON AJAX response leading to a JavaScript parse error, which prevented the overlay from disappearing and success message from showing up.

Here I'll provide more details on the issue and my findings...

<!-- excerpt_separator -->

### How The Injection Works

Based on my testing, the PHP Agent will inject the first part of the browser agent immediately after it detects the string "`<head>`" in the response body and the second part immediately before the string "`</body>`". This happens for any response with Content-Type of text/html, which is the default for PHP.

This can be demonstrated with this simple test case...

```php
<?php

echo '<head>HELLO</body>';
```

The injection happens **after** PHP has finished executing and preparing the response. This means that if the application sends back a JSON response that contains the string "`<head>`" and doesn't set an `application/json` Content-Type response header, New Relic will inject the browser agent JavaScript causing malformed JSON.

This can be demo-ed as follows...

```php
<?php

$response = array(
	'content' => '<html><head></head><body><h1>Hello</h1></body></html>'
);

echo json_encode($response);
```

Fortunately, this can be mitigated by simply setting the correct Content-Type for the JSON response, which is what we wound up doing for the client in this case...

```php
<?php

$response = array(
	'content' => '<html><head></head><body><h1>Hello</h1></body></html>'
);

header('Content-Type: application/json');
echo json_encode($response);
```

And the response...

```
{"content":"<html><head><\/head><body><h1>Hello<\/h1><\/body><\/html>"}
```

### My Thoughts On The Issue

While on one side of the argument, the application should just set the correct Content-Type for JSON responses, on the other side, New Relic could be a bit more cautious to prevent against this. For example, if the first byte of the response is a "`{`" it's probably a good idea to assume that the response is JSON and avoid injecting the browser agent, regardless of the `Content-Type`. 

I would argue that it's better to err on the side of caution here as I'd prefer to lose monitoring for some responses over completely breaking application functionality due to mangling of response bodies.

### Some Other Reference Points

Also, it's worth noting that this sample product case is not the only example of this behavior causing issues.

In fact, the [Aligent_NewRelicFix](https://github.com/aligent/Aligent_NewRelicFix) [Magento](https://magento.com/) module was built to fix the same issue which occurs when viewing transactional email templates in the Magento admin panel.

Further, there's [an additional issue in Magento 2 which can cause rendering of UI Components to fail](https://github.com/magento/up-for-grabs/issues/1). This issue was in fact even specifically addressed in version 7.6.0.201 of the New Relic PHP Agent.

> In some cases, Magento 2's UI Component JSON data sources return with an (incorrect) Content-Type of text/html. This resulted in auto instrumentation insertions into JSON data that contained a <head> string. We now identify these specific JSON data sources and ensure the auto instrumentation is not inserted into this JSON data.
> 
> [PHP Agent 7.6.0.201 Release Notes](https://docs.newrelic.com/docs/release-notes/agent-release-notes/php-release-notes/php-agent-760201)
